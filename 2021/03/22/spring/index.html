Spring基础使用

发表于2021-03-22|更新于2021-03-22|Spring
|阅读量:

1. 什么是Spring

Spring是一个轻量级的javaEE解决方案，整合了多种优秀的设计模式

  • 轻量级

    对于运行环境没有要求（开源：tomcat,resion。收费：weblogic …）

    代码的移植性高，不需要实现额外的接口

  • 设计模式

    工厂，代理，模板。

  • 控制反转(IOC Inverse of Control)

    把对于成员变量创建与赋值的控制权转交给Spring，从代码中转移到Spring工厂和配置文件中完成

  • 依赖注入(Dependency Injection DI)

    当一个类需要另一个类时，就意味着依赖，一旦出现依赖，就可以把另一个类作为本类的成员变量，最终通过Spring配置文件进行注入(赋值)。

2. Spring Hello World

环境：
1. JDK1.8+
2. Maven3.5+
3. IDEA2018+
4. SpringFramework 5.1.4

  1. 引入jar包

    <dependency>
  <groupId>org.springframework</groupId>
  <artifactId>spring-context</artifactId>
  <version>5.1.4.RELEASE</version>
</dependency>

  2. 编写配置

    1. 配置文件的放置位置：任意位置 没有硬性要求
2. 配置文件的命名   ：没有硬性要求  建议：applicationContext.xml
<bean id="person" class=" ... "/>

  3. 通过工厂类，获得对象

    ApplicationContext: Spring提供的工厂类对象
非web环境 ： ClassPathXmlApplicationContext (main junit)
web环境  ：  XmlWebApplicationContext

ApplicationContext cx = new ClassPathXmlApplicationContext("/applicationContext.xml");
Person person = (Person)cx.getBean("person");
//这样我们就通过spring工厂创建了一个对象

3. 注入

3.1 什么是注入

通过Spring工厂及配置文件，为所创建对象的成员变量赋值

3.2 为什么需要注入

通过编码的方式为成员变量赋值存在耦合，将这部分的赋值转移到配置文件中可以解耦合

衍生问题：之后我们更常用的是注解，不还是将赋值写在代码中了吗？

spring的配置优先于注解，如果对注解不满意，可以在配置文件中对其进行覆盖，也不用修改代码

3.3 如何注入

3.3.1 Set注入

spring通过调用对象的set方法完成成员变量的赋值

<bean id="user" class="cn.scarfbear.entity.User" >
    1. String + 8种基本类型    
        <property name="username">
        <value>scarfbear</value>
        <!-- 简写 <property name="username" value="scarfbear"></property> -->
        </property>
    2. 数组
        <property name="array">
        <list>
            <value>1</value>
            <value>2</value>
        </list>
        </property>

    <property name="list">
        <list>
            <value>1</value>
            <value>2</value>
        </list>

    4. set集合
    <property name="set">

            <value>2</value>

            </entry>
        </map>
    </property>
    6. Properties

3.4 类型转换器

  • 什么是类型转换器

    我们在使用注入是value中的值都是String字面量，注入到对象中却转换成了对应的类型，此时就是类型转换器在起作用。

    Spring提供了许多内置的类型转换器，比如String转8种基本类型，方便我们的使用。

    但仍不能囊括我们所有的需求，比如String转Data类型，因为各个地区对于data类型的书写习惯不同，(eg:2021/3/20,2021-3-20)。而官方只提供了2021/3/20类型转换器，如果想设置为自己习惯的其他格式，就需要我们自定义类型转换器。

  • 如何自定义类型转换器

    1. 实现Converter<from类型，to类型>接口
public class MyDateConverter implements Converter<String, Date> {
    //日期格式，在下一步中注入，方便扩展
    private String pattern;

    public void setPattern(String pattern) {
        this.pattern = pattern;
    }
    
    @Override
    public Date convert(String source) {

        Date date = null;
        try {
            SimpleDateFormat sdf = new SimpleDateFormat(pattern);
            date = sdf.parse(source);
        } catch (ParseException e) {
            e.printStackTrace();
        }
        return date;
    }
}

2. 工厂中配置自定义的类型转换器
<bean id="myDateConverter" class="xxx">
  <property name="pattern" value="yyyy-MM-dd"/>
</bean>

4. Spring创建对象

4.1 常规对象的创建

	注入相关代码
</bean>

4.2 复杂对象的创建

复杂对象：不能通过new关键字直接创建的对象。eg：Connection,SqlSessionFactory

4.2.1 实现FactoryBean接口

如果Class中指定的类型 是FactoryBean接口的实现类，那么通过id值获得的是这个类所创建的复杂对象
<bean id="conn" class="com.baizhiedu.factorybean.ConnectionFactoryBean"/>

如果想要获得FactoryBean对象本身则ctx.getBean("&conn")

4.2.2 实例工厂

好处：避免了Spring的侵入，整合遗留系统时使用。

<bean id="conn"  factory-bean="connFactory" factory-method="getConnection"/>

4.4.3 静态工厂

<bean id="conn" class="com.baizhiedu.factorybean.StaticConnectionFactory" factory-method="getConnection"/>

5. Spring创建对象的生命周期

一个对象创建，存活，消亡的过程。

5.1 创建过程

  • scope=”singleton”

    单例模式，工厂创建完成时就会创建对象

  • scope=”prototype”

    多例模式，从工厂里获取对象时，完成创建

5.2 初始化

Spring在创建完对象后，会调用对象的初始化方法，我们可以根据需求再初始化方法中完成自己需要的功能

如何自定义初始化方法？

  • 实现InitializingBean接口

    public class User implements InitializingBean {
    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("我初始化了");
    }
}

  • 在bean中将普通方法设为初始化方法

    public void myInit(){
  System.out.println("我初始化了");
}

<bean id="user" class="xxx" init-method="myInit"/>

5.3 销毁

Spring销毁对象前，会调用对象的销毁方法，完成销毁操作（只适用于单例）。

  • 何时销毁？

    工厂关闭时 ctx.close();

  • 自定义销毁方法

    同上

    //实现接口
public class User implements InitializingBean, DisposableBean {
    @Override
    public void destroy() throws Exception {
		System.out.println("我被销毁了了");
    }
}

//在bean中将普通方法设为初始化方法
<bean id="user" class="xxx" destroy-method="myDestroy"/>

6. Aop

Aop(Aspect Oriented Programming) 面向切面编程

  • 为什么要使用Aop?

    在写业务类的时候，有一些和业务代码无关的额外功能，例如日志记录，事务控制等功能，很多类都需要。然而直接写在业务类中既累赘又容易与业务代码混淆，此时就需要用到Aop。

    Spring使用代理模式解决这个问题，我们把业务代码写在原始类中，然后再创造一个代理类将原始类与额外功能整合起来，使用时使用代理对象。

6.1 静态代理（不用）

为所有的原始类手动编写一个代理类

  • 为什么不用？

    文件过多不好管理

    维护和修改麻烦

6.2 动态代理

解决了静态代理的通点，并且实现了代理功能

  • 所需依赖

    <dependency>
  <groupId>org.aspectj</groupId>
  <artifactId>aspectjrt</artifactId>

<dependency>

6.2.1 MethodBeforeAdvice接口

  1. 编写额外功能（只能在前面添加额外功能，感觉用处不大）

    public class Before implements MethodBeforeAdvice {
    @Override
    public void before(Method method, Object[] objects, Object o) throws Throwable {
        System.out.println("这里是额外功能，以下是参数");
        System.out.println("method:" + method);
        System.out.println("objects:" + objects);
        System.out.println("o:" + o);
    }
}

  2. 编写额外功能的切入点，并将切入点与额外功能组合起来

    <aop:config>
    <aop:pointcut id="pc" expression="execution(* *(..))"></aop:pointcut>
    <aop:advisor advice-ref="before" pointcut-ref="pc"/>
</aop:config>

切入点表达式：
*  *(..)  --> 所有方法

* ---> 修饰符 返回值
* ---> 方法名 可以带详细的包名与类名
()---> 参数表
..---> 对于参数没有要求 (参数有没有，参数有几个都行，参数是什么类型的都行)

6.6.2 MethodInterceptor

  1. 编写额外功能

    public class Arround implements MethodInterceptor {
    @Override
    public Object invoke(MethodInvocation invocation) throws Throwable {
        System.out.println("在前面添加额外功能");
        Object proceed = invocation.proceed();
        System.out.println("在后面添加额外功能");
        return proceed;
    }
}

  2. 编写额外功能的切入点，并将切入点与额外功能组合起来

    <aop:advisor advice-ref="arround"
             pointcut="execution(* cn.scarfbear.service.*.*(..))"></aop:advisor>

7.2 Spring控制事务

7.2.1 事务属性

@Transactional(isloation= 1. 隔离属性
               propagation= 2. 传播属性
               readOnly= 3. 只读属性
               timeout= 4. 超时属性
               rollbackFor= 5. 异常属性 回滚
               noRollbackFor= 5. 异常属性 提交
              )

7.2.2 隔离属性

他描述了事务解决并发问题的特征

  1. 有哪些并发问题

    • 脏读

      一个事务，读取了另一个事务中没有提交的数据。会在本事务中产生数据不一致的问题

      解决方案 隔离级别设置为读已提交 @Transactional(isolation=Isolation.READ_COMMITTED)

    • 不可重复读

      一个事务中，多次读取相同的数据，但是读取结果不一样。会在本事务中产生数据不一致的问题

      解决方案 隔离级别设置为可重复读 @Transactional(isolation=Isolation.REPEATABLE_READ)
      本质： 一把行锁

    • 幻读

      一个事务中，多次对整表进行查询统计，但是结果不一样，会在本事务中产生数据不一致的问题
      解决方案 隔离级别设置为串行化 @Transactional(isolation=Isolation.SERIALIZABLE)
      本质：表锁

  2. 数据库默认的隔离级别

    MySQL : REPEATABLE_READ
    Oracle: READ_COMMITTED

    平常使用默认就足够了

7.2.3 传播属性

他描述了事务解决嵌套问题的特征

  1. 什么是事务的嵌套

    Service调用Service时，会发生事务中还有多个小事务。当内层事务被回滚时，外层事务会怎样？传播属性就是解决事务的嵌套的

  2. 传播属性的值及其用法

    传播属性的值 外部不存在事务 外部存在事务 用法 备注
    REQUIRED 开启新的事务 融合到外部事务中 @Transactional(propagation = Propagation.REQUIRED) 适用增删改方法 （默认的隔离级别）
    SUPPORTS 不开启事务 融合到外部事务中 @Transactional(propagation = Propagation.SUPPORTS) 适用查询方法
    REQUIRES_NEW 开启新的事务 挂起外部事务，创建新的事务 @Transactional(propagation = Propagation.REQUIRES_NEW) 日志记录方法中
    NOT_SUPPORTED 不开启事务 挂起外部事务 @Transactional(propagation = Propagation.NOT_SUPPORTED) 及其不常用
    NEVER 不开启事务 抛出异常 @Transactional(propagation = Propagation.NEVER) 及其不常用
    MANDATORY 抛出异常 融合到外部事务中 @Transactional(propagation = Propagation.MANDATORY) 及其不常用
  • REQUIRED： 增删改查方法肯定是需要一个事务来包裹着的，因此外面不存在事务时需要新建一个事务来包裹，当外层有事务时则融合到外层事务中，不干扰其他事务。
  • SUPPORTS： 查询方法则不太需要事务包裹，因此外面不存在事务时则什么也不做，当外部存在事务时则融合，不干扰其他方法
  • REQUIRES_NEW：日志本质上是增加一条日志行。因此没事务肯定需要新建一个事务，外部有事务时需要挂起外部事务，这里的挂起理解为暂停外部事务，以内部事务为优先。因为日志操作，就算你操作出现问题被回滚，但操作记录还是应该被记录下来。因此应以日志事务为最高优先级，先提交日志事务，再去操作其他事务。

7.2.4 只读属性

针对于只进行查询操作的业务方法，可以加入只读属性，提供运行效率

默认： false

7.2.5 超时属性

指定了事务等待的最长时间

  1. 为什么事务进行等待？
    当前事务访问数据时，有可能访问的数据被别的事务进行加锁的处理，那么此时本事务就必须进行等待。
  2. 等待时间 ：秒
  3. 超时属性的默认值 -1
    表示最终由对应的数据库来指定

7.2.6 异常属性

Spring事务异常的处理
默认 对于RuntimeException及其子类 采用的是回滚的策略
默认 对于Exception及其子类 采用的是提交的策略

  1. 使用方式

    rollbackFor = {java.lang.Exception,xxx,xxx}
    noRollbackFor = {java.lang.RuntimeException,xxx,xx}

    推荐使用默认配置，自定义RuntimeException与Exception的子类

9. Spring注解方式开发

  • 开发环境搭建

    <context:component-scan base-package="cn.scarfbear"/>
作用：让Spring框架在设置包及其子包中扫描对应的注解，使其生效。

或者在配置类上使用@ComponentScan("cn.scarfbear")注解

9.1 Spring工厂相关注解

<bean id="user" class="cn.scarfbear.entity.User" scope="prototype" lazy-init="true"
	init-method="" destroy-method="">

  1. 在类上，多用于自己开发的类上

    bean标签使用@Component替代，相对应的还有衍生注解
    @Repository —> XXXDAO @Service —> XXXService @Controller —> xxxController

    id默认为类名小写，可在@Component中Value属性中自定义id
    scope 对应 @scope(“xxx”)注解
    lazy-init 对应 @Lazy注解
    init-method 对应 @PostConstruct注解
    destroy-method 对应 @PreDestroy注解

  2. 在方法上，多用于整合其他框架

    bean标签使用@Bean注解替换

    id默认为方法名，可在value中自定义。

9.2 Spring注入相关注解

  1. @Autowired【推荐】【注入自定义类型】

    基于类型的注入：注入对象的类型，必须与目标成员变量类型相同或者是其子类（实现类）

    放置位置：成员变量或其set方法上

  2. @Autowired+@Qualifier

    基于名字的注入：注入对象的id值，必须与Qualifier注解中设置的名字相同

  3. @Value【注入基本类型】

    可以直接在Value中写值

  4. @Value+@PropertySource

    @PropertySource中标明Properties文件位置，使用@Value(“${key}”)来引用文件中配置的值
